# v2.0 发布说明

> 文档版本：v2.0
>
> 版本定位：v1.0 之后的新增能力白皮书。v2.0 接替 v1.0 继续描述 Foggy 语义层引擎能力，但不把 v1.0 标记为废弃，也不替代 v1.0 的基础语法参考。
>
> 实现口径：本文以 Java 引擎已经实现并经过签收的能力为主，不把规划项、prompt-only 评估或未签收实验能力写成稳定功能。

## 核心变化

v1.0 证明了面向 LLM 的语义层引擎基础：TM/QM 语义建模、JSON Query DSL、MCP 工具、权限治理、查询证据和 Compose 组合式分析。

v2.0 的重点是把“可治理查询”推进到“可治理复杂分析”：

- Java MCP 可以在路由校准后重新分发，而不是继续执行过期计划。
- 查询过程可以输出非阻断 trace，便于回放 catalog、planner、tool call、validator 和 query result。
- DSL_CTE 形成分阶段分析执行面，支持受治理聚合、派生、窗口、后置过滤、排序和有限跨结果对齐。
- relation result-stage 表达式扩展为一组签名小公式，包括比率、差额、差异率、绝对差异率、标签和有序分桶。
- Memory Grid 从组合式分析概念推进到有句柄、生命周期、资源上限和 fail-closed guard 的受控二次分析边界。
- Pivot 透视能力进入 Java 引擎签收范围，覆盖树形轴、轴窗口、下钻合同和多数据库日期口径修正。
- Experience Recipe 从离线经验方案推进到最小 registry lookup，保留 namespace、tenant、permission、owner 和 active 状态过滤。

## 新增能力摘要

| 能力域 | v2.0 新增内容 | 边界 |
|---|---|---|
| Java MCP 编排 | calibrated-route re-dispatch、非阻断 trace correlation | 不等于自治分析师；仍依赖 validator、权限和 fail-closed |
| DSL_CTE 分阶段分析 | 聚合、派生、窗口、postSlice、orderBy 和受控桥接模板 | 不开放任意 SQL CTE 或物理表 Join |
| 受治理表达式 | ratio、difference、delta ratio、absolute delta ratio、single-threshold label、ordered bucket、alias DAG 分层、签名 ranking | 不开放任意函数、任意 CASE、物理字段引用或自由表达式语言 |
| Memory Grid | bounded result handle、store-backed resolver、生命周期、资源限制、跨模型 alignment contract | 不开放无界明细、任意 DuckDB API、full outer、多 key join 或长期 durable backend 承诺 |
| Pivot | 树形轴、axis window、axis domain selection、cascade drilldown、weekday dialect parity | 不开放未签名 shape、任意 interactive expand/collapse 或完整前端透视产品 |
| Experience Recipe | 最小 registry exact lookup、治理过滤、发布门证据引用和签名 verifier SPI | 不等于完整 recipe 市场、向量检索、管理后台或生产信任根 |
| 多数据库证据 | SQLite、MySQL8、PostgreSQL 的 pivot/DSL_CTE 证据；SQL Server weekday parity 证据 | 不声明完整 SQL Server Maven profile release gate |

## 能力状态口径

v2.0 文档使用三类状态描述能力：

| 状态 | 含义 | 文档写法 |
|---|---|---|
| 已签收 | Java runtime、validator、fixture、CI 或验收记录已经覆盖该能力 | 可以作为 v2.0 稳定能力描述 |
| 窄合同可用 | 能力可用，但只在 signed template、特定模型、特定 shape 或特定方言证据内成立 | 必须同时写适用场景和拒绝边界 |
| 后续方向 | 有规划或评估材料，但尚不能作为稳定 runtime 能力承诺 | 只能写入“不纳入 v2.0”或后续方向，不写成可用功能 |

## 与 v1.0 的关系

v2.0 继承 v1.0 的基本原则：

- LLM 不直接访问裸数据库 schema。
- 请求通过语义模型、结构化 DSL 和工具协议进入引擎。
- 权限、字段校验、方言转换和执行证据由引擎承担。
- 不把提示词约束当成安全边界。

v2.0 新增的是更高阶的执行合同和可观测能力，而不是放松治理边界。复杂分析能力越强，越需要更明确的 signed contract、测试证据和关闭式失败。

## 未纳入 v2.0 的能力

以下内容不作为 v2.0 稳定能力发布：

- 自由物理 SQL 执行。
- 任意 Semantic SQL、任意 CTE 或任意表达式语言。
- 通用跨模型 Join、通用漏斗、通用归因或任意时期对比。
- 完整 BI、报表设计器、SQL notebook 或客户可见 AI 分析工作台。
- 完整 SQL Server release gate、生产级 durable Memory Grid backend、完整 recipe 市场或向量检索治理。
- Python 与 Java 对所有新增能力的完全同步 parity。
